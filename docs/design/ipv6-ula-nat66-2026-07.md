## 1. Problem & approach

We want IPv6 on the homelab LAN. Two viable shapes exist on a UDM Pro Max:

- **(A) PA-prefix-on-LAN** â€” advertise the KPN delegated GUA prefix
  (`2a02:a46d:10d2::/48`) directly onto the VLANs via SLAAC.
- **(B) ULA-on-LAN + NAT66 to the active WAN** â€” give each VLAN a stable,
  ISP-independent ULA and source-translate egress to the WAN's GUA.

We choose **(B)**. Reasons, all grounded in the research doc:

1. **UniFi cannot route IPv6 across dual WAN.** Assign-only; traffic does not
   follow the surviving WAN and per-WAN IPv6 traffic rules are ignored â€”
   re-confirmed by community â‰ˆMay 2026, no UI-Team response, no release note
   contradicts it (research Â§Q3, Dual-WAN thread). So a PA-on-LAN design is
   *married to whichever single WAN currently assigns the prefix*, and on that
   WAN's PD changing (KPN can and does reshuffle) every internal address,
   firewall rule and any AAAA record churns.
2. **Renumbering independence.** ULA (`fd00::/8`) is ours forever, independent
   of KPN or Ziggo. Internal addressing, firewall rules and host-to-host
   traffic stay stable across any ISP PD change. Native since Network 10.0.160
   ("Additional IPs" incl. ULA) and NAT66 since 9.4.17 â€” both present on our
   box, no 10.5-specific dependency (research Â§1, ULA/NPTv6 threads).
3. **Dual-WAN later is additive, not a re-address.** Because the LAN carries a
   fixed ULA, adopting a second WAN's v6 later means "add a second NAT66
   mapping + source policy", with the LAN untouched (see Â§6). PA-on-LAN would
   force a full renumber to do the same.

Cost of (B): NAT66 on UniFi is **stateful** (the GUI itself warns performance
may be degraded â€” research Â§1); v6 sessions are not preserved across a WAN
change. Acceptable for a home LAN. We single-home v6 to **KPN only** now
(Ziggo does not currently delegate a routable PD â€” only an IA_NA link /128,
`ipv6_enabled` not even armed; research Â§Q4 + memory).

## 2. Addressing plan

**ULA /48 (RFC 4193 Â§3.2.2):** `fdd5:5b90:1c27::/48`

Generated 2026-07-08 by the RFC-prescribed method: 64-bit NTP timestamp +
node EUI + extra `os.urandom` entropy â†’ SHA-1 â†’ least-significant 40 bits =
global ID `d5:5b90:1c27`, prefixed with `fd`. Not sequential, not a
well-known example prefix. (Generation script recorded in the change record.)

**Per-VLAN /64 layout** â€” subnet id = the VLAN's number written as literal hex
digits, so the /64 reads like the VLAN it belongs to:

| VLAN | ID | IPv4 (unchanged) | ULA /64 | Gateway (UDM) |
|---|---|---|---|---|
| CLIENT-VLAN | 1 (native) | 172.16.2.0/23 | `fdd5:5b90:1c27:1::/64` | `fdd5:5b90:1c27:1::1` |
| SERVER-VLAN | 10 | 172.16.10.0/23 | `fdd5:5b90:1c27:10::/64` | `fdd5:5b90:1c27:10::1` |
| GUEST-VLAN | 200 | 192.168.2.0/24 | `fdd5:5b90:1c27:200::/64` | `fdd5:5b90:1c27:200::1` |
| IOT-VLAN | 250 | 172.16.4.0/23 | `fdd5:5b90:1c27:250::/64` | `fdd5:5b90:1c27:250::1` |

(The literal digits `10`/`200`/`250` are hex values `0x10`/`0x200`/`0x250`;
they are distinct and unique â€” readability, not arithmetic, is the goal.)

**Phase-1 scope â€” CLIENT-VLAN only.** The original ask was CLIENT-VLAN, and it
is the right first target: it is the busiest dual-stack VLAN, easiest to
observe, and its failure mode (happy-eyeballs falls back to v4) is benign.
SERVER-VLAN, IOT-VLAN, GUEST-VLAN are pre-allocated above but **not enabled in
phase 1**:

- **SERVER-VLAN** â€” defer until the cluster's v6 posture is designed separately
  (Cilium/k8s v6 is its own project; do not surprise it with RAs).
- **IOT-VLAN** â€” many IoT devices have poor/again-buggy v6 stacks; low value,
  real risk. Defer.
- **GUEST-VLAN** â€” Hotspot zone, isolated; enable only if guests need v6. Defer.

## 3. NAT66 rule (single-WAN, KPN)

**Intent:** source-NAT (masquerade) all LAN ULA egress to the KPN WAN's GUA.

- **Match:** source `fdd5:5b90:1c27::/48` (the whole ULA /48, so enabling more
  VLANs later needs no new NAT rule), egress interface = **KPN WAN**
  (eth9 / ppp, PPPoE VLAN 6, `wan_failover_priority=1`).
- **Action:** translate source to the KPN delegated prefix
  (base `2a02:a46d:10d2::/48`). UniFi models this as a Policy-Table **NAT66**
  rule with a "Translated IP Address" set to the delegated prefix (research Â§1).
- **Single-WAN by design:** *no* Ziggo NAT66 rule in phase 1. On KPN outage,
  v6 egress stops (IPv4 fails over to Ziggo; happy-eyeballs keeps clients
  usable). This is the accepted tradeoff (research option (c)).
- NAT66 is **not** a security boundary â€” it hides nothing that the firewall
  isn't already denying. Inbound protection comes from Â§5, not from NAT.

**Drive-mechanism / exact fields:** [per Iris affordance probe â€” in progress].
Whether the NAT66 rule and the "Translated IP = delegated prefix" binding are

is likely the rule references the delegated prefix indirectly. If GUI-only,

## 4. RA / SLAAC + DNS

**Address assignment: SLAAC, ULA-only.**
- RA on CLIENT-VLAN advertises `fdd5:5b90:1c27:1::/64`, `A`-flag set
  (SLAAC), **`M`=0** (no DHCPv6 address), default router = UDM link-local.
- **No GUA on the LAN.** Critical: if a GUA is ever advertised alongside the
  ULA, RFC 6724 makes clients prefer the GUA and the NAT66 intent is defeated
  (research Â§3). ULA-only keeps all egress on the translated path.
- **Conservative RA lifetimes during rollout** (e.g. preferred 600s / valid
  1800s) so that rollback (Â§7) drains client ULAs in minutes, not hours. Raise
  to normal lifetimes once stable.

**DNS â€” recommended: keep DNS transport on IPv4 in phase 1; advertise NO v6 DNS.**
- Clients already have a working v4 resolver (the UDM). They resolve external
  **AAAA** records over that v4 transport perfectly well and then connect over
  v6. There is no need to hand a v6 resolver address to get v6 connectivity.
  This removes RDNSS/DHCPv6-DNS as a moving part (`O`-flag=0). Smaller blast
  radius; fewer client-OS quirks (old Android has no DHCPv6).
- **Internal `*.bluejungle.net` stays IPv4-only.** These are per-hostname
  **A** records on the UDM written by external-dns (`external-dns-unifi`
  memory). We add **no AAAA**. This is a hard rule, not a preference:
  **AAAA writes to the UniFi `static-dns` API are a documented footgun that
  crashed the UDM on 2026-05-13** (`unifi-aaaa-static-dns-footgun` memory).
  Consequence: internal service traffic remains v4 end-to-end, which is also
  exactly what shrinks this change's blast radius â€” v6 is *egress-only* in
  phase 1.
- Alternative (deferred): advertise the UDM ULA (`fdd5:5b90:1c27:1::1`) as an
  RDNSS resolver later if a v6-only client ever appears. Not needed now.

## 5. Firewall

Target posture mirrors the existing v4 zone model, for v6:

- **WAN inbound (KPN): default-deny.** No unsolicited inbound v6. Only
  established/related return traffic for LAN-initiated flows. This is the real
  inbound protection (NAT66 provides none â€” Â§3).
- **Egress:** allow CLIENT-VLAN ULA â†’ internet (v6), stateful return.
- **Inter-VLAN parity:** CLIENT/SERVER/IOT are the "Internal" zone (inter-VLAN
  routing allowed for v4 today); GUEST is "Hotspot" (isolated). The v6 rules
  must reproduce this: Internal-zone v6 inter-VLAN permitted, GUEST v6 isolated
  with no inter-VLAN. In phase 1 only CLIENT is v6-enabled so inter-VLAN v6 is
  moot, but the rule set is authored to parity so enabling SERVER/IOT later
  needs no firewall rewrite.
- **Dynamic-prefix-proof inbound rules:** should we *ever* need an inbound v6
  pinhole, key it on the **IPv6 Interface ID** (supported since the NPTv6 work,
  research Â§1) not on the GUA, so KPN PD reshuffles don't break it. Phase 1 has
  **zero** inbound pinholes.

## 6. Switch-path to dual-WAN (first-class requirement)

The whole reason for ULA is that going dual-homed later is **additive**. When
UniFi ships routable dual-WAN v6 (or if we accept the scripted-failover hack
from research Â§2), the minimal change is:

1. **LAN unchanged.** No client re-addressing, no RA change, no DNS change,
   no firewall rewrite â€” the ULA and all rules key on the ULA /48 and interface
   IDs, never on a WAN GUA.
2. **Confirm Ziggo actually delegates a routable PD** (today it does not: IA_NA
   link /128 only, `ipv6_enabled` unset; research Â§Q4 says line-dependent â€”
   former-Ziggo=/56 routable, former-UPC=DS-Lite/CGNAT). Arm `ipv6_enabled` on
   Ziggo WAN and verify a real IA_PD arrives.
3. **Add a second NAT66 rule:** source `fdd5:5b90:1c27::/48` â†’ masquerade to the
   Ziggo delegated prefix, egress = Ziggo WAN. (Mirror of Â§3.)
4. **Add source/WAN selection** so v6 follows the active WAN. This is the piece
   UniFi does not do natively today â€” either the GUI primitive if/when shipped,
   or the udm-boot-wrapped `ip -6 rule` + `ip6tables MASQUERADE` daemon
   (research Â§2), accepting session-drop-on-flip and the auto-update fragility
   noted there.

That is the entire delta. Nothing in Â§2â€“Â§5 is revisited.

## 7. Risk / rollback

**Risk tier: HIGH.** Touches WAN-adjacent NAT66 + firewall zones + LAN
addressing (RA). Requires explicit `user` approval + Themis QA pass, freeze
check, and the test-plan+evidence in Â§8. `sensitive`-adjacent (the WAN/gateway).

**Blast radius.** IPv6 is net-new on every VLAN today (all
`ipv6_interface_type=none`), so there is no existing v6 to break. Phase 1 is
CLIENT-VLAN only; all IPv4 is untouched; internal DNS stays v4 (no AAAA).
Realistic failure modes:
- **Client dual-stack stall.** If v6 is advertised but NAT66 black-holes egress,
  happy-eyeballs *should* fall back to v4 within ~250ms; a mis-scoped RA could
  still cause perceptible stalls. Mitigated by short RA lifetimes + the Â§8 test
  before broad exposure.
- **UDM instability from stateful NAT66 load** (GUI's own perf warning). Watch
  gateway CPU/health after enabling.
- **RFC 6724 ULA-precedence quirk.** Some OSes historically rank ULA below IPv4
  and would then *not* use v6 externally. Â§8 test #3 is the go/no-go on this.
- **The AAAA/static-dns crash** â€” avoided by design (Â§4), called out so no one
  "helpfully" adds AAAA later.

**Rollback (fast, complete):**
1. Delete the NAT66 rule (Â§3).
2. Set CLIENT-VLAN back to `ipv6_interface_type: none` â€” RAs stop; clients drain
   their ULA within the (short) valid lifetime.
3. Delete the v6 firewall rules (Â§5).
Because internal DNS never received AAAA and internal traffic never left v4,
this restores the exact prior state with no residue. Capture a read-back
confirming CLIENT-VLAN is `none` again.

## 8. Validation plan (end-to-end, user-visible) â€” required, Themis gate

Executed by Iris on a **real client** on CLIENT-VLAN (e.g. a laptop or the
Mac Mini agent host, if on CLIENT-VLAN). Evidence artefacts attached to the
change record; Themis reviews the evidence, not just the diff.

1. **Client gets a ULA (no GUA).** `ip -6 addr` (or `ipconfig` on Windows) shows
   an address in `fdd5:5b90:1c27:1::/64` + link-local, **and no** global
   `2a02:...` address on the interface. Default v6 route â†’ UDM link-local.
   *Evidence:* command output.
2. **AAAA resolution over v4 transport.** `dig AAAA ipv6.google.com` returns
   AAAA records. *Evidence:* dig output.
3. **Egress + source translation (the key test).**
   `curl -6 https://ipv6.icanhazip.com` (or `https://api6.ipify.org`) returns a
   **public** address, and that address is **inside KPN's
   `2a02:a46d:10d2::/48`** â€” proving the client egressed over v6 with its source
   masqueraded to the KPN GUA (NOT the ULA). Cross-check the returned prefix
   against the live KPN delegation. Also run the browser score at
   `https://test-ipv6.com` and capture it via `kiosk-verify` (it is a rendered
   UI surface). *Evidence:* curl output showing the `2a02:a46d:10d2:*` source +
   test-ipv6.com screenshot.
4. **Internal stays v4 / no AAAA leak.** `dig AAAA nas.bluejungle.net` returns
   **no** AAAA; `https://nas.bluejungle.net` still works (over v4). *Evidence:*
   dig output + reachability check.
5. **UDM read-back (verify-before-report).** GET the CLIENT-VLAN network object
   and the NAT66 rule back after the write and confirm they match intent (UniFi

6. **Single-WAN behaviour documented (observe-only).** Note that on KPN outage

   to test (disruptive); state it as the accepted phase-1 tradeoff.

If test #3 shows the client egressing over IPv4 instead of v6 (the RFC 6724
quirk), the change is **not** complete â€” record the observed OS behaviour and
